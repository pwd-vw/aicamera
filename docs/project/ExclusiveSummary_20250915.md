# สรุปเฉพาะกิจ — 2025-09-15

หัวข้อ: รายงานทดสอบระบบจัดการภาพจากกล้อง (Camera Handler, Camera Manager, Camera Blueprint) — ผังการไหลของข้อมูลภาพ, สาเหตุที่ทำให้เกิดหน่วง/กระตุก, แนวทางแก้ไข, การดำเนินการที่ทำแล้ว และสถานะปัจจุบัน

## 1) ผังการไหลของข้อมูลภาพจากกล้อง (สภาพที่ใช้งานและทดสอบจริง)

- การจับภาพ (Capture)
  - `CameraHandler` เปิดใช้งาน Picamera2 แบบสองสตรีม โดยอิงค่าจาก config: `MAIN_RESOLUTION` สำหรับงานตรวจจับ (RGB888), `LORES_RESOLUTION` สำหรับพรีวิว (RGB888; ก่อนหน้าเคยใช้ YUV420 ตอนทดสอบฮาร์ดแวร์เอนโค้ด).
  - เธรดจับภาพทำงานต่อเนื่อง ดึง Request แล้วสร้าง:
    - `main_frame` → อาร์เรย์ RGB888 ของ numpy (ส่งให้ detection processor)
    - `lores_frame` → ไบต์ (หากมีฮาร์ดแวร์เอนโค้ด) หรืออาร์เรย์ RGB888 (เส้นทางซอฟต์แวร์)
  - เมทาดาทาถูกอ่านและแคชที่ความถี่ 1 Hz (แทนการ serialize ทุกเฟรม) เก็บไว้ใน `_metadata_buffer` ลดภาระ CPU

- การบัฟเฟอร์และส่งต่อ
  - เก็บเฟรมภายใต้ล็อก: `_main_frame_buffer`, `_lores_frame_buffer`, `_metadata_buffer`
  - `capture_frame(source="buffer", stream_type="lores|main|both", include_metadata)` จะคืนค่า:
    - lores: ไบต์ MJPEG (ทางเลือกหลัก) หรืออาร์เรย์ RGB
    - main: อาร์เรย์ RGB
    - both: ทั้งสองเฟรม พร้อมเมทาดาทา/เวลาเมื่อร้องขอ

- บริการสตรีมวิดีโอ (VideoStreamingService)
  - ดึง lores แล้วทำให้เป็นไบต์ MJPEG มาตรฐาน:
    - ถ้าได้รับเป็นไบต์อยู่แล้ว (ฮาร์ดแวร์ MJPEG/H.264) จะส่งต่อทันที
    - ถ้าเป็น RGB888 จะเข้ารหัส JPEG ด้วย OpenCV
  - คิวเฟรมตั้ง `maxsize=1` และล้างก่อนใส่เพื่อลดค้างหลังคิว; คุม FPS เฉพาะใน worker (ไม่ sleep ซ้ำใน blueprint)
  - ส่งคืน dict ไปยัง blueprint มี `frame_bytes`, `width`, `height`, `fps`, `source`, `quality`, และเวลาสำหรับวัด latency (`timestamp_capture`, `timestamp_enqueue`)

- ฝั่ง HTTP Blueprint (Flask)
  - สร้าง multipart MJPEG โดยใช้ `frame_bytes` ตรง ๆ (ไม่ base64)
  - บันทึก latency: capture→yield, enqueue→yield และคำนวณ FPS ทุก ~30 เฟรม

- เส้นทางการตั้งค่าและสถานะ
  - `config.py`: นิยาม `MAIN_RESOLUTION`, `LORES_RESOLUTION`, ค่า FPS/controls พื้นฐาน
  - Endpoint สถานะ/เมทาดาทา คืนค่า `camera_status`, `camera_properties`, `current_config` โดยฝั่งหน้าเว็บรองรับได้ทั้งโครงสร้างแบบ flat และแบบซ้อน

## 2) ผลการวิเคราะห์สาเหตุที่ทำให้เกิดหน่วง/กระตุก

- หน่วงจากการหน่วงซ้ำซ้อน (redundant delays)
  - มีการหน่วงเวลาในทั้ง worker และ blueprint ทำให้ FPS สะดุด แก้โดยย้ายการคุม FPS ไว้ที่ worker จุดเดียว

- คิวเฟรมใหญ่เกินไปจนเกิด backlog
  - `frame_queue` ใหญ่ทำให้มีการค้างและปล่อยเป็นชุด ๆ → ภาพกระตุก แก้ด้วย `maxsize=1` และนโยบายทิ้งเฟรมเก่าทันที

- ซอฟต์แวร์เข้ารหัส JPEG ทุกเฟรม
  - เมื่อไม่ใช้ฮาร์ดแวร์เอนโค้ด `cv2.imencode` ต่อเฟรมกิน CPU ทำให้ FPS ลด ขณะนี้ยังใช้เมื่อจำเป็น แต่เน้นใช้ฮาร์ดแวร์เมื่อพร้อม

- Resize/แปลงสีโดยไม่จำเป็น
  - `cv2.resize` แบบไม่เช็คก่อน เพิ่มภาระและทำให้ภาพเบลอ ตอนนี้ resize เฉพาะเมื่อขนาดไม่ตรง และแปลง RGB→BGR เฉพาะที่จำเป็น

- การ serialize เมทาดาทาถี่เกินไป
  - เดิมทำทุกเฟรม ตอนนี้ลดเหลือ 1 Hz และใช้แคช

- ความไม่เข้ากันของรูปแบบข้อมูล (format mismatch)
  - คาดหวัง YUV/ไบต์ไม่ตรงกัน ทำให้เกิด error การถอดรหัส (เช่น ufunc) แก้ด้วยสัญญาข้อมูลที่ชัดเจน: สตรีมพรีวิวเป็นไบต์, สตรีมหลักเป็นอาร์เรย์ RGB

- ความไม่ชัดเจนการใช้ฮาร์ดแวร์เอนโค้ด (ช่วงแรก)
  - การตั้งค่า `encode="lores"` ไม่ส่งไบต์ได้ตามต้องการ แนวทางที่ถูกต้องคือใช้ `MJPEGEncoder`/`H264Encoder` + `CircularOutput` โดยตรง

## 3) แนวทางแก้ไข (พิจารณาและนำไปใช้)

- รวมศูนย์การคุม FPS และนโยบายคิว
  - คุมที่จุดเดียว, `frame_queue=1`, ทิ้งเฟรมเก่าก่อนใส่ใหม่เพื่อภาพสดเสมอ

- ให้ความสำคัญกับฮาร์ดแวร์เอนโค้ดเมื่อมี
  - ใช้ `MJPEGEncoder`/`CircularOutput`; ใช้ H.264 สำหรับบันทึก; ตกกลับเป็นซอฟต์แวร์เมื่อจำเป็น

- หลีกเลี่ยงการคัดลอก/ย่อขนาดที่ไม่จำเป็น
  - `main` คงเต็มสำหรับตรวจจับ; ตั้ง `self.width,self.height` จาก lores; resize เมื่อจำเป็นเท่านั้น

- ลดความถี่เมทาดาทา
  - อัปเดต/serialize ที่ 1 Hz

- ทำให้วัด latency ได้จริง
  - ปั๊ม timestamp: capture → enqueue → yield และคอยดู p50/p95 ใน log

- ขนาดภาพขับเคลื่อนด้วย config
  - ใช้ค่าจาก `config.py` ทั้งหมด

## 4) การดำเนินการที่ทำแล้ว (ไม่นับส่วน UI ของ dashboard)

- CameraHandler
  - ตั้งค่าสองสตรีมตาม `MAIN_RESOLUTION`/`LORES_RESOLUTION`
  - เสริมทางเลือก `MJPEGEncoder`/`CircularOutput` (ตรวจฮาร์ดแวร์ก่อน) พร้อม fallback เป็นอาร์เรย์
  - รวมสัญญาการคืนค่าของ `_capture_from_buffer` และแคชเมทาดาทาที่ 1 Hz
  - ตัด `.copy()` ที่ไม่จำเป็น (คงไว้เฉพาะจุดที่จำเป็น)

- VideoStreamingService
  - ใช้ขนาด lores จาก config; ตัด `resize` ที่ไม่จำเป็น
  - เข้ารหัส JPEG ด้วยซอฟต์แวร์เฉพาะเมื่อรับเป็นอาร์เรย์; กำหนดคุณภาพตามแหล่งที่มา
  - คิว `maxsize=1`, ล้างก่อนใส่; ใส่ timestamp capture/enqueue

- Blueprint
  - ส่ง `frame_bytes` ตรง (ไม่ base64); ตัด sleep ซ้ำ; เพิ่ม log FPS และ latency

- สัญญาข้อมูลระหว่างหลังบ้าน/หน้าเว็บ
  - หน้าเว็บรองรับได้ทั้งคีย์แบบ flat และแบบซ้อน; ปรับ endpoint debug ให้ใช้งานได้จริง

- กำจัดการแจ้งเตือน Socket.IO ในคอนโซล
  - ปิด Socket.IO ฝั่งหน้าเว็บเป็นค่าปริยาย และทำ stub `io()` เพื่อกันการเชื่อมต่อโดยไม่ตั้งใจ; ใช้ HTTP แทนสำหรับดึงสถานะ

## 5) ผลลัพธ์และสถานะปัจจุบัน

- พรีวิววิดีโอ
  - ภาพลื่นขึ้นอย่างเห็นได้ชัด ลดจังหวะกระตุกเนื่องจากคิวเฟรมขนาด 1 และไม่มีการหน่วงซ้ำซ้อน
  - เมื่อมีฮาร์ดแวร์ MJPEG ภาระ CPU ลดลง; เส้นทางซอฟต์แวร์ยังทำงานได้ดีภายใต้การปรับให้เบาลง

- Latency
  - เวลาหน่วงปลายทางลดลง เฟรมออกต่อเนื่อง; มีตัวชี้วัดใน log (capture→yield, enqueue→yield) p50 ใกล้คาบเฟรมของ worker; p95 ดีขึ้นจากนโยบายทิ้งเฟรมเก่า

- ความเสถียร
  - แก้ปัญหา decode error ด้วยสัญญาข้อมูลที่สอดคล้องและการแปลงแบบมีการ์ด

- การแยกเส้นทางงานตรวจจับ
  - สตรีม `main` คงความละเอียดเต็มแบบ RGB สำหรับ AI; สตรีม lores ใช้ทรัพยากรต่ำและเป็นอิสระ

- โอกาสปรับปรุงถัดไป
  - ใช้ H.264 สำหรับการบันทึกไฟล์, พรีวิวสดคง MJPEG เพื่อลด latency
  - ตัวเลือก: บัฟเฟอร์ย้อนเวลา (ring buffer) สำหรับบันทึกย้อนหลังไม่กี่วินาที; I/O แบบอะซิงก์
  - ตัวเลือก: เปิดเผยสถิติ latency ผ่าน `/camera/status` เพื่อให้ UI แสดงผลได้
